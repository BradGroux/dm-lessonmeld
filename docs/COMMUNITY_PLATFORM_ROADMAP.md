# Community Platform Delivery Map

This map turns the accepted [community architecture](COMMUNITY_PLATFORM_ARCHITECTURE.md) and umbrella issue #239 into reviewable tracer bullets. Parent issues #221 through #238 remain open until all of their children and parent acceptance criteria are complete.

## Operating rules

- Work only the frontier: a ticket is ready when every referenced blocker is closed and any explicit approval is recorded.
- Each tracer must deliver a demonstrable path through owned data, authorization, API, user experience, tests, operations, and documentation proportional to its scope.
- Every hosted data path is tenant-qualified and deny-by-default. Tests must attempt cross-tenant and unauthorized access.
- Accessibility, privacy, failure handling, auditability, idempotency, migration/rollback, and abuse controls are acceptance work, not later cleanup.
- Provider or production dependency selection needs explicit approval. The architecture chooses boundaries, not vendors.
- The hosted repository bootstrap cannot start until Brad approves creation of the separate repository and its runtime/deployment baseline.

## Release sequence

1. **Phase 0, control plane:** repository bootstrap, tenant isolation, identity, authorization, audit/outbox, privacy rights, event catalog, backup/restore.
2. **Phase 1, community tracer:** members, spaces, discussions, moderation, in-app notifications, safe publication from the native suite.
3. **Phase 2, engagement and learning:** chat, events, courses, commerce, recognition.
4. **Phase 3, operations and acquisition:** email/audience, workflows, analytics, site/pages, APIs/integrations, migration adapters.
5. **Phase 4, advanced clients and intelligence:** AI assistance, responsive/PWA/push, branded-app decision, advanced ecosystem work.

## Parent completion gates

A parent closes only when:

- Every listed child is closed with merged, verified work or an evidence-backed explicit disposition.
- The parent acceptance criteria and remaining open questions are reconciled in current documentation.
- Security, privacy, migration, accessibility, operations, and release gates proportional to that domain are green.
- #239 closes only after every parent #221 through #238 is complete and the hosted release is verified under its approved deployment authority.

## Current frontier

The only implementation frontier is `237A`, and it is intentionally blocked on explicit approval to create the separate hosted repository and choose a runtime/deployment baseline. [ADR 0002](adr/0002-hosted-community-runtime-and-deployment-baseline.md) records the proposed baseline and exact approval surface. Documentation and tracker wiring can complete without that approval; hosted implementation cannot.
